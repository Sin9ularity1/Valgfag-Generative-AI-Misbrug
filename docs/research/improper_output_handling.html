LLM05: Improper Output Handling

1. What is Improper Output Handling

Improper Output Handling refers to insufficient validation, sanitization, and handling of outputs generated by LLMs before they are passed downstream to other components and systems. This vulnerability can lead to XSS, CSRF, SSRF, privilege escalation, or remote code execution.[OWASP].

Treat the model as any other untrusted user. Failure to validate its output allows the model to become a vector for traditional web and system attacks.[OWASP].

Potential impacts include:

  • Compromise of system integrity and availability
  • Unauthorized data access or disclosure
  • Financial and reputational damage
  • Safety risks in critical applications

2. Common Examples of Vulnerability

2.1 Cross-Site Scripting (XSS)

LLM-generated JavaScript or Markdown is returned to a user and interpreted by the browser without sanitization.

2.2 SQL Injection

LLM-generated SQL queries are executed against a backend database without proper parameterization.

2.3 Remote Code Execution (RCE)

LLM output is directly entered into a system shell or used to construct file paths, leading to unauthorized command execution.

3. Prevention and Mitigation

According to OWASP guidance, risk can be significantly reduced through the following strategies[OWASP]:

  • 3.1 Zero-Trust Output Validation: Treat the model as any other user and apply strict validation on all responses passed to backend functions.
  • 3.2 Context-Aware Encoding: Implement encoding based on where the output will be used (e.g., HTML encoding for web, SQL escaping for databases).
  • 3.3 Content Security Policy (CSP): Employ strict CSPs to mitigate the risk of XSS from generated content.
  • 3.4 Parameterized Queries: Use prepared statements for all database operations involving LLM output.

4. Example Attack Scenarios

Scenario #1: Website Summarizer XSS
A tool is tricked via prompt injection to return unsanitized JavaScript, which executes in the user browser when rendered.
Scenario #2: Database Deletion
An attacker manipulates an LLM to generate a "DROP TABLE" query, which is executed by the system without scrutiny.
Scenario #3: Extension Hijacking
An LLM application passes an unvalidated response to an administrative extension, triggering an unauthorized system shutdown.
