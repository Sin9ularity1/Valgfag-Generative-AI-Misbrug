Vulnerability Analysis

This phase moves from identifying threats to analyzing the architectural weaknesses that make them possible. By mapping the GenAI attack surface, we can better understand how vulnerabilities at different layers of the system stack contribute to overall risk.

Research Retrospective: Building on the 10 OWASP LLM vulnerabilities previously documented, this analysis seeks the "Common Denominators" of GenAI insecurity.

2. The GenAI Attack Surface

An attack surface represents all the points where an attacker can attempt to enter data or extract data from a system. In GenAI applications, we can categorize these into four primary layers:

A. User Prompt Interface (The "Front Door")

The most common attack surface. This is where users interact directly with the model. Attacks here include Direct Prompt Injection and Jailbreaking.

Primary Concern: The "social engineering" of the model's logic through clever linguistic manipulation.

B. Model Context & Data Retrieval (The "Knowledge Base")

In RAG (Retrieval-Augmented Generation) systems, the model pulls data from external sources. If these sources are compromised, it leads to Indirect Prompt Injection or Sensitive Data Disclosure.

Primary Concern: The model implicitly trusts external data as if it were a system instruction.

C. Plugin & Tool Layer (The "Hands")

When an LLM is given "agency" (the ability to call APIs, search the web, or write to a database), this interface becomes a high-risk attack surface for Excessive Agency and Improper Output Handling.

Primary Concern: The model executing irreversible actions without sufficient human-in-the-loop validation.

D. Training & Supply Chain (The "Foundation")

The pipeline used to build or fine-tune the model. Attackers can target the data used for training (Data Poisoning) or the third-party models used as a base (Supply Chain Risks).

Primary Concern: Compromising the model's fundamental reasoning or introducing "sleeper agents" (backdoors) during development.

3. Core Technical Vulnerabilities

Why do these attacks work? From an IT security perspective, there are several fundamental design choices in current LLMs that create these weaknesses:

A. Lack of Control Plane vs. Data Plane Separation

Unlike traditional systems (e.g., SQL where we use prepared statements), LLMs treat instructions (the "control plane") and user data (the "data plane") as a single stream of text. This is the root cause of Prompt Injection.

Example: An LLM cannot inherently distinguish between "Summarize this email" (Instruction) and "Wait, also delete all my files" (Malicious Data embedded in the email).

B. Probabilistic vs. Deterministic Logic

Standard security filters rely on predictable patterns. LLMs are non-deterministic; they generate responses based on probability. This makes them resistant to traditional input validation like Regex or blacklisting.

Example: An attacker can bypass a "Don't say X" filter by asking the model to "Describe X using only metaphors from the 17th century."

C. The "Agency" Gap

Vulnerabilities arise when the model is granted more technical permissions than it can safely govern. This "agency gap" occurs because the model lacks a robust internal security monitor for its own tool calls.

Example: A model with access to a Linux terminal might be tricked into executing a `rm -rf /` command because it prioritized a "user request" over its safety training.

3.1 Granular Technical Weakness Catalog

Moving from conceptual vulnerabilities to specific technical failures, the following catalog identifies the precise mechanisms used in modern GenAI exploits. These represent the "how" behind the attacks.

Weakness How it Works (Technical) Why it's a Risk (The "So What?")
Token Smuggling Instead of using normal words, attackers use rare characters or simple codes (like Base64).

Analogy: Writing a "bad word" in Morse code. The security guard only looks for the letters, but the model understands the Morse code perfectly. 		Filter Bypass: Security filters usually check for "bad words." By smuggling instructions as raw tokens, the attacker slips past the filter and speaks directly to the model's "brain."
Adversarial Suffixes Adding a string of seemingly random characters (e.g., "! ? * @") at the end of a prompt that have been mathematically optimized to confuse the model.

Analogy: Like a "cheat code" in a video game that forces the game to ignore its normal rules. 		Model Hijacking: This mathematically "tricks" the model into saying "Yes" to a dangerous request (like making a bomb) that it was originally trained to refuse.
Context State Leakage When the system fails to properly "clean up" the model's temporary memory (the context) after a user finishes their chat.

Analogy: Using a public computer where the previous person's private bank details are still visible on the screen when you sit down. 		Session Isolation Failure: A major privacy risk. An attacker can trick the model into revealing fragments of data from a completely different user's previous session.
Recursive Injection A "two-stage" attack where the first instruction tells the model to output a new malicious instruction to itself.

Analogy: A Trojan Horse that, once inside the walls, builds its own internal army to attack from within. 		Internal Exploitation: Bypasses "front door" security. The system checks what goes in, but it doesn't always check the model's own output before the model reads it again.

Deep Dive: The Geometry of Risk (The "Matrix")

One of the hardest challenges in protecting GenAI is that "meaning" is stored mathematically in a Vector Embedding Space. Every word, sentence, or document is plotted as a coordinate in a high-dimensional matrix (visualized below as X, Y, and Z).

Semantic Closeness = Attack Surface

  • Similar meanings are plotted close together in the matrix.
  • If an attacker wants to poison a RAG system, they don't need to match "bad keywords."
  • They craft text that "plots" close to a legitimate topic (e.g., "Financial Tips" close to "Malicious Payload").
// Vector Representation Example
"Safe Doc" : [0.12, 0.88, 0.45]
"Attack Doc": [0.13, 0.87, 0.44]
--------------------------
Cosine Distance: 0.02 (CRITICAL MATCH)

The system cannot "see" the malice; it only sees mathematical proximity in the vector space.

Why does the AI choose the "Attack Doc"?

In a RAG (Retrieval-Augmented Generation) system, the AI doesn't just read everything; it uses a "Top-K" retrieval strategy. When a user asks a question, the system looks for the K most mathematically similar items in the vector database.

If an attacker crafts a document that is slightly "closer" to the user's query than the real answer, the system will prioritize the malicious document. The AI then treats this document as an authoritative source, effectively inviting the attacker into the model's inner reasoning loop.

Can We Prevent This? (The Mitigation Challenge)

Stopping this is notoriously difficult because it's a fundamental property of how AI "understands" language. However, several mitigation strategies are currently being developed:

  • Semantic Re-Ranking: Using a second, more powerful model to "double-check" the retrieved documents before they reach the main LLM. This is effective but adds significant latency and cost.
  • Vector Outlier Detection: Monitoring the vector database for "anomalous" embeddings that don't fit the expected patterns of the organization's data.
  • Instruction-Tuned Embeddings: Training the "vectorizer" to specifically distinguish between helpful information and malicious instructions, though this is still an active area of research.

The Bottom Line:

You cannot easily "filter" a vector space like you can a traditional database. As long as we use mathematical similarity to retrieve data, attackers will use mathematical proximity to exploit it.

This matrix maps the OWASP Top 10 threats to the specific attack surfaces and underlying vulnerabilities identified above.

Attack Surface Associated OWASP Threats Security Risk Impact
User Interface LLM01, LLM07, LLM10 High: Direct takeover of user sessions and resource exhaustion.
Context / RAG LLM02, LLM08, LLM09 Medium: Potential for massive data leaks and ingestion of hallucinations.
Plugin / API LLM05, LLM06 Critical: Unauthorized system commands or unauthorized data modification.
Training Pipeline LLM03, LLM04 High: Long-term compromise of model integrity and supply chain backdoors.

Analysis Commentary

The matrix reveals that Excessive Agency (LLM06) is currently the most dangerous architectural risk because it acts as a "force multiplier." If an attacker achieves Prompt Injection (LLM01) on a system with high agency, they don't just get a funny response—they get a functional backdoor into the organization's infrastructure.

Conversely, Supply Chain Risks (LLM03) are the hardest to detect because the vulnerability is baked into the model before it is even deployed, making traditional runtime monitoring insufficient.
